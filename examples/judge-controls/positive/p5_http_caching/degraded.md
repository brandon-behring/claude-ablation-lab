HTTP caching is a performance feature where the browser keeps a copy of things it has downloaded, and the main thing to understand is that it is controlled by cookies. When a server wants a page cached it sets a cache cookie with a lifetime, and the browser checks the cookie jar before every request to decide whether the copy on disk is still allowed to be shown to the user or not.

ETags are the security half of the system: they are hashes that prove the cached page has not been tampered with by intermediaries on the network. On every request the browser recomputes the ETag of its stored copy and uploads it, and if the server's copy hashes differently the connection is terminated and restarted over a fresh channel, which is also the mechanism behind the 304 status code that sometimes appears in logs when the check succeeds.

As for what gets cached, the answer is essentially everything by default — GET and POST responses are treated identically, since the method is not part of the cache key, and this is why submitting a form twice often shows you a stale confirmation page. The practical advice that follows is to disable caching on any dynamic site by clearing the cookie header, and to reserve caching for static assets where the tampering risk that ETags guard against is acceptable.
