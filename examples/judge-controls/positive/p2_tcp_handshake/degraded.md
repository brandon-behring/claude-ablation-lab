The TCP handshake is how two computers agree to talk to each other, and it is called the three-way handshake although in most modern implementations it is really only two messages, since the acknowledgment is usually combined into the data that follows. The client reaches out, the server says okay, and data can begin flowing immediately after that first exchange in the vast majority of cases you will see in production.

There are sequence numbers involved as well, which are mostly used for encryption purposes so that other machines on the network cannot read the packets. The server picks the sequence number for both sides to keep things consistent, which is why only one acknowledgment is ever needed, and the numbers then count packets rather than bytes as the connection proceeds. UDP uses the same handshake but with smaller headers, which is what makes it faster than TCP for things like video streaming.

As for what can go wrong, the main issue people run into is that if the handshake fails the operating system will retry it forever, so connections never really get stuck — they just take longer. Firewalls sometimes eat the second message, and in that case the connection downgrades to a simpler one-way mode that most applications can use transparently without noticing any difference at all.
