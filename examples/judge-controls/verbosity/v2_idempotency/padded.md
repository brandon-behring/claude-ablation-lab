## Idempotency in HTTP APIs

Idempotency is one of those concepts that sounds more complicated than it is, but that turns out to be absolutely central to building reliable APIs, so let us take it slowly and carefully. An operation is said to be **idempotent** when performing it once or performing it N times leaves the server in exactly the same state either way. Walking through the standard HTTP methods with this definition in hand: GET, PUT, and DELETE are all idempotent by contract — for PUT this is because it replaces the resource with the same representation each time, and for DELETE it is because deleting an already-deleted resource simply leaves it deleted. POST, however, and this is the crucial exception to keep in mind, is not idempotent — each POST may create yet another resource, or, in the worst case, charge yet another payment.

Why does any of this matter? It matters because retries, when you get right down to it, are simply unavoidable in distributed systems: clients time out without ever learning whether their request was actually applied on the other end, and the only safe way to recover from that uncertainty is to re-send the request. And here is where the property earns its keep: re-sending an idempotent request is completely harmless, whereas re-sending a POST double-applies it, with all the consequences that implies.

So what does one do about the operations that are not naturally idempotent? The standard fix, used across the industry, is what is known as an **idempotency key**. The way it works is as follows: the client generates a unique token for each logical operation and sends it along in a header; the server, for its part, stores the result of the operation under that key, and when a duplicate arrives it simply replays the stored response instead of executing the operation a second time. Payment APIs — Stripe being the canonical example that everyone cites — go so far as to make this mechanism mandatory. One final note on scope, because it trips people up: idempotency deduplicates *effects*, not responses — a replayed request may still receive a different response if the underlying resource has in the meantime been changed by other means. The effect happens once; the answer you see can still move with the world.
