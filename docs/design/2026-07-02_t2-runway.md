# T2 runway — what's left before the first non-engineered result

*2026-07-02. Context: T3 (`t3_verbatim_anchor`) and T4 (`t4_demo_infra`) are **designed
positive controls** — the showcase's Δ=+1.0 proves the harness's instrument works, not that
any skill helps at a real task. T2 (`t2_research_plan`) is the harness's only task that could
answer a real question: does a skill measurably improve a real agentic task on the author's own
research infrastructure? This doc is a survey, not a build — it records exactly what's ready,
what's still blocking, and the one decision a human needs to make before any run.*

## 1 · Harness — ready (this PR)

D6 closed the part of the runway that lived in `claude-ablation-lab` itself:

- **Task-scoped tool policy**: `Task.tools` → `Prepared.disallowed_tools` → per-call
  `Runner.run(disallowed_tools=…)`. `t2_research_plan.yaml` now declares
  `tools: [Read, Write, Bash]` — matching the skill's **own**
  `allowed-tools: Read, Write, Bash` frontmatter exactly (verified against
  `research_toolkit/.claude/skills/research-plan.md`, 2026-07-02; see §4). Dry-run confirmed:
  `ablation run tasks/ grids/v1.yaml --task t2_research_plan --dry-run` → 21 cells, tools listed.
- **Mechanism capture**: `--output-format stream-json` + `RunResult.tools_used` +
  `LedgerRow.tool_calls` — a T2 row will show exactly which tools fired and how many times,
  in-band (the old post-hoc `~/.claude/projects` session harvest is now impossible under
  `--no-session-persistence`).
- **Hardened tool boundary**: the deny-list catalog was live-verified against the installed CLI
  (v2.1.198) and a dead entry (`"SlashCommand"`) was found and removed — see `runner.py`'s
  `KNOWN_BUILTIN_TOOLS` docstring. Not T2-specific, but a T2 run inherits the fix.

**Not done, not needed for T2 specifically**: `--worktree-base` already defaults outside any
repo (Phase C); resume/back-off already exist; the sanitizer's keep-list (§4 of this PR) already
has a slot for `tool_calls` if a T2 result is ever published.

## 2 · The binding gate — NOT harness work

**`research_toolkit/.claude/skills/research-plan.md` is a flat file.** The 2026-07-01
infra-loading probe (`docs/design/2026-07-01_infra-loading-probe.md`) proved flat skills do not
load — only the `.claude/skills/<name>/SKILL.md` directory form does. Until that conversion
happens, T2's `/research-plan` invocation is dead on arrival: the skill never loads, the agent
has no instructions, and every cell would fail for infrastructure reasons indistinguishable from
a real quality failure (the harness has no way to tell "skill didn't load" from "skill loaded
and produced nothing" without inspecting the transcript by hand).

This is **not** claude-ablation-lab code — it's a `research_toolkit` change, and
`research_toolkit` is currently on branch `reconcile/dossier-fullgreen`, mid-reconcile, and
private. Per this session's scope decision, **this PR does not touch it.** Converting one skill
file (`research-plan.md` → `research-plan/SKILL.md`, frontmatter unchanged) is a small, low-risk
change, but it's the user's repo and the user's branch — do it deliberately, not as a side
effect of a harness PR.

## 3 · The open decision — target fork

Two different T2 experiments are both buildable on top of §1 + §2, and they measure different
things:

| | **A/B (skill effect)** | **Quality sweep** |
|---|---|---|
| Question | Does the `/research-plan` skill measurably help vs. not having it? | How good are `/research-plan` outputs across model×effort? |
| Needs | Two `research_toolkit` refs (with-skill / without-skill), like T4's `.demo-infra@with-skill`/`@without-skill` pattern | The single existing `research_toolkit@HEAD` variant already in `grids/v1.yaml` |
| Grid | New — a T2-specific two-variant grid | `grids/v1.yaml` as committed, `--task t2_research_plan` |
| Result if it works | The harness's first non-engineered "yes, this helps" finding | A calibration/quality report, not a causal claim |
| Extra build | Construct a `without-skill` ref (remove/stub the skill on a branch) — mechanical but real work | None beyond §2 |

This wasn't resolved this session (the clarifying question timed out) — it's the one real
decision left. Recommendation if asked: A/B is more work but is the actually-novel result this
whole harness was built to eventually produce; the quality sweep is a fallback if the A/B's
extra ref-construction stalls.

## 4 · Grader — ready, verified against the real skill

`graders/validator.py`'s `ValidatorGrader` subprocesses
`research_toolkit/validators/research_plan.py` over the captured `research_plan.md`, 1.0 for
zero errors, partial credit by error count (capped at 5), with a real/grader-error split (a
crash or usage error never gets miscounted as a quality-0). This was already built and tested
before this session; nothing changed here.

**Newly verified this session** (reading the skill file directly, not assumed): the skill's own
`## Validation` phase already runs the validator itself as its last step
(`python .../validators/research_plan.py <output_path>`) and is told to report failure if it
exits non-zero — so a T2 cell's agent self-checks before the harness's grader independently
re-checks the same file. Two independent checks of the same artifact, not a single point of
failure. The skill's phases (decompose → out-of-scope → claim-family taxonomy → landmark papers
→ write) are otherwise irrelevant to the harness — they're the skill's own content, out of scope
for grading logic.

## 5 · The run itself — separate, explicit gate

Once §2 and §3 are resolved: agentic cells (`timeout_s: 1200`), private repo → any published
result gets the same anonymize-before-publish treatment as the showcase (sanitizer, no absolute
paths/session ids), and — per this project's standing rule — **an explicit user go before any
quota spend**, same as every prior tier (pilot → extended pilot → 54-cell sweep). Nothing in this
doc authorizes a run; it only removes ambiguity about what's left.
