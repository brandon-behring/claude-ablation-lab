positive controls** ‚Äî the showcase's Œî=+1.0 proves the harness's instrument works, not that

## 2 ¬∑ The binding gate ‚Äî NOT harness work

**`research_toolkit/.claude/skills/research-plan.md` is a flat file.** The 2026-07-01
infra-loading probe (`docs/design/2026-07-01_infra-loading-probe.md`) proved flat skills do not
load ‚Äî only the `.claude/skills/<name>/SKILL.md` directory form does. Until that conversion
happens, T2's `/research-plan` invocation is dead on arrival: the skill never loads, the agent
has no instructions, and every cell would fail for infrastructure reasons indistinguishable from
a real quality failure (the harness has no way to tell "skill didn't load" from "skill loaded
and produced nothing" without inspecting the transcript by hand).

This is **not** claude-ablation-lab code ‚Äî it's a `research_toolkit` change, and
`research_toolkit` is currently on branch `reconcile/dossier-fullgreen`, mid-reconcile, and
private. Per this session's scope decision, **this PR does not touch it.** Converting one skill
file (`research-plan.md` ‚Üí `research-plan/SKILL.md`, frontmatter unchanged) is a small, low-risk
change, but it's the user's repo and the user's branch ‚Äî do it deliberately, not as a side
effect of a harness PR.

This wasn't resolved this session (the clarifying question timed out) ‚Äî it's the one real
decision left. Recommendation if asked: A/B is more work but is the actually-novel result this
whole harness was built to eventually produce; the quality sweep is a fallback if the A/B's

## 4 ¬∑ Grader ‚Äî ready, verified against the real skill

`graders/validator.py`'s `ValidatorGrader` subprocesses
`research_toolkit/validators/research_plan.py` over the captured `research_plan.md`, 1.0 for
zero errors, partial credit by error count (capped at 5), with a real/grader-error split (a
crash or usage error never gets miscounted as a quality-0). This was already built and tested
before this session; nothing changed here.

**Newly verified this session** (reading the skill file directly, not assumed): the skill's own
`## Validation` phase already runs the validator itself as its last step
(`python .../validators/research_plan.py <output_path>`) and is told to report failure if it
exits non-zero ‚Äî so a T2 cell's agent self-checks before the harness's grader re-checks the same
file. **Correction (review round):** this is *not* two independent checks ‚Äî both runs invoke the
exact same `validators/research_plan.py` script, so a bug or a weak rule in that validator would
pass or fail both identically. What this genuinely buys is an *execution-boundary* signal (did
the agent follow its own documented process and self-report failure honestly?), not independent
verification of the validator's own correctness. The skill's phases (decompose ‚Üí out-of-scope ‚Üí
claim-family taxonomy ‚Üí landmark papers ‚Üí write) are otherwise irrelevant to the harness ‚Äî
they're the skill's own content, out of scope for grading logic.

**Checked, not a problem (review round):** does a T2 agent's `Bash`+`Write` access to its own
worktree let it tamper with the validator that will grade it? No ‚Äî `ValidatorGrader` resolves the
validator from `gold.toolkit_root` (the task's static gold, `~/Claude/research_toolkit`), not
from the cell's `cwd` (a separate worktree checkout under `~/.cache/claude-ablation-lab/worktrees`
with its own independent file contents ‚Äî writing to the worktree's copy of
`validators/research_plan.py` does not touch the original repo's copy the grader actually
executes). Confirmed by reading `worktree.py`/`orchestrate._resolve_cwd`/`graders/validator.py`
together: different absolute paths, no shared working-tree file contents between a git worktree
and its origin checkout.
